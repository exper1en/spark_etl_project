# Аналитический пайплайн обработки пользовательских событий на Apache Spark

## Обзор
В рамках проекта реализован полноценный конвейер обработки данных (ETL) с использованием Apache Spark. На вход подаётся сгенерированный датасет с кликами пользователей, содержащий как корректные, так и намеренно испорченные записи. Выполняются последовательные шаги: извлечение, очистка, трансформация, агрегация и сохранение результатов в удобном для анализа виде.

## Организация файлов
- `data/` – исходные данные (`clickstream.csv`), создаваемые скриптом-генератором.
- `src/` – исходный код:
  - `generate_data.py` – утилита для генерации тестового набора (55 000 строк, из них 10% содержат дефекты).
  - `etl_pipeline.py` – основной модуль, реализующий полный цикл обработки.
  - `verify_results.py` – скрипт для проверки корректности выходных данных.
- `output/` – каталог с результатами работы (очищенные данные, агрегаты, отчёты). Каждый запуск создаёт подкаталог с временной меткой.
- `docs/` – документация и скриншоты.
- `requirements.txt` – список зависимостей.

## Необходимые компоненты
- Python версии 3.11 (или 3.13 при условии совместимости пакетов)
- Java Development Kit 11
- Apache Spark 3.5.8

## Инструкция по развёртыванию и запуску

1. **Получение кода**
   ```bash
   git clone https://github.com/your-username/spark-etl-project.git
   cd spark-etl-project
   ```

2. **Настройка виртуального окружения**
   ```bash
   python -m venv venv
   source venv/bin/activate   # для Linux/Mac
   .\venv\Scripts\activate    # для Windows
   pip install -r requirements.txt
   ```

3. **Запуск Spark-кластера (standalone)**
   - Откройте два терминала (PowerShell).
   - В первом запустите мастер:
     ```bash
     cd C:\spark\spark-3.5.8-bin-hadoop3
     .\bin\spark-class.cmd org.apache.spark.deploy.master.Master
     ```
   - Во втором – воркер, указав адрес мастера (обычно `spark://localhost:7077`):
     ```bash
     cd C:\spark\spark-3.5.8-bin-hadoop3
     .\bin\spark-class.cmd org.apache.spark.deploy.worker.Worker spark://localhost:7077
     ```

4. **Генерация тестовых данных**
   ```bash
   cd src
   python generate_data.py
   ```

5. **Запуск пайплайна**
   ```bash
   python etl_pipeline.py
   ```

6. **Проверка результатов**
   Убедитесь, что в папке `output/` появилась новая директория с временной меткой, содержащая все ожидаемые файлы.

## Детальное описание работы

### Генерация данных (`generate_data.py`)
Для создания синтетического набора используются библиотеки `pandas`, `numpy` и `faker`. Из 55 000 записей 90% являются «чистыми» (корректные форматы, положительные значения, существующие регионы), а оставшиеся 10% содержат различные аномалии: пустые идентификаторы, отрицательные длительности, некорректные даты, неизвестные типы устройств и т.п. Это позволяет проверить устойчивость конвейера к «грязным» данным. Результат сохраняется в `data/clickstream.csv` (UTF-8).

### ETL-процесс (`etl_pipeline.py`)
Главный класс `SparkETLPipeline` управляет всеми этапами.

#### Инициализация SparkSession
Создаётся подключение к мастеру Spark с заданным URL. Устанавливаются параметры, улучшающие работу на Windows: используется специальный коммиттер для Parquet, игнорируются ошибки очистки временных файлов, увеличиваются таймауты сетевых соединений. При необходимости можно поднять уровень логирования до `ERROR`, чтобы подавить многочисленные предупреждения.

#### Извлечение (`extract`)
Чтение CSV-файла выполняется с указанием схемы. Включён режим `PERMISSIVE` и колонка `_corrupt_record` – это позволяет Spark помечать строки, не соответствующие типам полей. Если такая колонка отсутствует, значит, все строки формально подходят под схему, но это не гарантирует отсутствия логических ошибок (например, пустых строк в обязательных полях). Дальнейшая очистка обработает их.

#### Трансформация (`transform`)
- **Очистка:** удаляются дубликаты по ключевым полям, пропуски заменяются значениями по умолчанию, отсеиваются записи с пустым `user_id`, отрицательной длительностью или ценой.
- **Обогащение:** из строкового поля `timestamp` извлекаются компоненты даты и времени (`event_date`, `event_hour`, `event_dayofweek`). Сессии классифицируются по длительности (`short`, `medium`, `long`), а цены – по диапазонам (`low`, `medium`, `high`).

В результате количество записей сокращается с 55 000 до примерно 53 144 – отбрасывается около 1 856 «грязных» строк.

#### Агрегация (`analyze`)
Вычисляются четыре набора статистик:
- **Активность по регионам и часам** – количество событий, уникальных пользователей, средняя длительность, суммарная выручка.
- **Статистика по устройствам** – распределение сессий по типам устройств и категориям длительности.
- **Топ-100 пользователей** – рейтинг по числу сессий (реализован без оконной функции, чтобы избежать предупреждений).
- **Ежедневная активность** – общее число событий, пользователей и выручка за каждый день.

#### Загрузка (`load`)
Для каждого запуска создаётся отдельная папка output/run_<timestamp> с подкаталогами:

- cleaned_data/clickstream_cleaned – очищенные данные в формате Parquet, партиционированные по дате.

- aggregated/ – агрегаты в Parquet.

- reports/ – CSV-отчёты (по одному файлу на каждый тип агрегации).

- execution_report.txt – текстовый отчёт со статистикой и примерами строк.

После записи всех данных выполняется принудительное удаление временных папок _temporary, которые Spark создаёт в процессе работы. Это гарантирует, что на диске не остаётся «мусора».

Проверка результатов (verify_results.py)
Скрипт запускает локальную сессию Spark и проверяет наличие всех ожидаемых файлов. Выводит количество записей в каждом наборе и несколько примеров строк, подтверждая корректность работы пайплайна.

#### О предупреждениях в консоли
При выполнении вы можете наблюдать сообщения вида:

```text
WARN FileUtil: Failed to delete file or dir ... it still exists.
WARN TransportChannelHandler: Exception in connection ...
```
Это предупреждения, а не фатальные ошибки. Они возникают из-за особенностей работы Spark на Windows:

- WARN FileUtil – Spark пытается удалить временные файлы, которые уже были удалены нами вручную. Предупреждение не означает, что мусор остался – мы сами позаботились об очистке.

- WARN TransportChannelHandler – возникает при завершении сессии, когда соединения между драйвером и воркерами закрываются. Это стандартное поведение и не влияет на уже сохранённые данные.

Если подобные сообщения мешают, можно повысить уровень логирования до ERROR (раскомментировав соответствующую строку в коде), но тогда исчезнут и другие полезные предупреждения.

#### Выводы
Разработанный конвейер демонстрирует полный цикл обработки данных на Apache Spark: от генерации сырых логов до получения структурированных аналитических витрин. Особое внимание уделено устойчивости к некорректным записям и автоматической очистке временных файлов. Все предупреждения, возникающие в процессе выполнения на Windows, не свидетельствуют о проблемах с данными и могут быть проигнорированы или скрыты сменой уровня логирования.
